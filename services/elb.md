# ELB (Elastic Load Balancing)

## 概要

複数のサーバーにトラフィックを分散させるロードバランサーサービス。Auto Scalingと組み合わせて高可用性・スケーラビリティを実現する。

---

## 種類

### ALB (Application Load Balancer) ★よく使う

- **レイヤー7（HTTP/HTTPS）**で動作
- URLパスやホスト名でルーティング先を分けられる
- WebアプリやAPIに最適
- WebSocketサポート
- gRPCサポート

```
/api/*  → バックエンドサービスのターゲットグループ
/web/*  → フロントエンドサービスのターゲットグループ
```

### NLB (Network Load Balancer)

- **レイヤー4（TCP/UDP）**で動作
- 超低レイテンシ・高スループット
- 固定IPアドレスを持てる
- ゲームサーバー、IoT、リアルタイム通信に向く

### GWLB (Gateway Load Balancer)

- ファイアウォールや侵入検知システム（IDS/IPS）のデプロイに使う
- あまり一般的ではない

### CLB (Classic Load Balancer)

- 旧世代。新規利用は非推奨

---

## ALBの主要概念

```
ALB
 └─ リスナー（ポート80, 443など）
      └─ ルール（パス/ホストでルーティング）
           └─ ターゲットグループ（EC2・コンテナ・Lambdaのグループ）
```

| 概念 | 説明 |
|---|---|
| **リスナー** | どのポート/プロトコルで受け付けるか |
| **ルール** | リクエストをどのターゲットグループに送るか条件を定義 |
| **ターゲットグループ** | 振り分け先のインスタンス群。ヘルスチェックもここで設定 |

---

## ヘルスチェック

ターゲットグループのインスタンスが正常か定期的に確認。

- 不健全なインスタンスへはトラフィックを送らない
- Auto Scalingと連携して不健全なインスタンスを自動交換

---

## スティッキーセッション

同じユーザーのリクエストを同じインスタンスに送り続ける機能（Cookie使用）。

セッション情報をElastiCacheやDBに保存するステートレス設計が推奨なので、極力使わないほうがいい。

---

## セキュリティ

- ALBはパブリックサブネットに配置し、EC2はプライベートサブネットに置くのが基本
- Security GroupでALBからの通信のみEC2に許可する設定が一般的
- ACMでHTTPS証明書を管理（ALBで終端）

---

## よく使う構成

```
インターネット
    ↓
ALB（パブリックサブネット・複数AZ）
    ↓ 分散
EC2 / ECS タスク（プライベートサブネット・複数AZ）
    ↓
RDS / ElastiCache
```
