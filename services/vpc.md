# VPC (Virtual Private Cloud)

## 概要

AWS上に作る仮想プライベートネットワーク。自分専用のネットワーク空間を定義して、EC2やRDSなどをその中に配置する。

---

## 主要コンポーネント

```
VPC (例: 10.0.0.0/16)
  ├─ パブリックサブネット (10.0.1.0/24)   ← インターネットから到達可能
  │    ├─ EC2 (Webサーバー)
  │    └─ ALB
  │
  ├─ プライベートサブネット (10.0.2.0/24)  ← インターネットから直接到達不可
  │    ├─ EC2 (アプリサーバー)
  │    └─ RDS
  │
  ├─ インターネットゲートウェイ (IGW)       ← VPCとインターネットの出入口
  ├─ NATゲートウェイ                       ← プライベートサブネットからの外向き通信
  └─ ルートテーブル                         ← パケットの経路を定義
```

---

## CIDR（IPアドレス範囲）

VPCやサブネットはCIDR記法でIPアドレス範囲を指定する。

```
10.0.0.0/16  → 10.0.0.0 〜 10.0.255.255（65,536アドレス）
10.0.1.0/24  → 10.0.1.0 〜 10.0.1.255（256アドレス）
10.0.1.0/28  → 10.0.1.0 〜 10.0.1.15（16アドレス）
```

- VPCの `/16` を大きい単位として確保し、サブネットを `/24` 単位で切り出すのが一般的
- サブネットの先頭4つと最後の1つはAWSが予約しているので使えない（/24なら実質251個）

---

## サブネット

VPC内をさらに分割したネットワーク。**1つのAZに紐付く**（複数AZにまたがれない）。

| 種類 | 特徴 |
|---|---|
| **パブリックサブネット** | IGWへのルートがあり、インターネットと通信可能 |
| **プライベートサブネット** | インターネットへの直接ルートなし。NATゲートウェイ経由で外向き通信は可能 |

高可用性のため、複数のAZにサブネットを作るのが基本。

```
AZ-a: パブリックサブネット(10.0.1.0/24) + プライベートサブネット(10.0.2.0/24)
AZ-c: パブリックサブネット(10.0.3.0/24) + プライベートサブネット(10.0.4.0/24)
```

---

## インターネットゲートウェイ (IGW)

VPCとインターネットを繋ぐゲートウェイ。VPCに1つアタッチする。

- パブリックサブネットのルートテーブルに `0.0.0.0/0 → IGW` を追加することでインターネットと通信できる
- IGWは冗長化・スケーリングが自動で行われるためユーザーの管理不要

---

## ルートテーブル

サブネット内のパケットをどこに送るかを定義する経路表。

| 送信先 | ターゲット | 意味 |
|---|---|---|
| `10.0.0.0/16` | local | VPC内の通信 |
| `0.0.0.0/0` | IGW | それ以外はインターネットへ（パブリック用） |
| `0.0.0.0/0` | NAT GW | それ以外はNAT経由（プライベート用） |

---

## NATゲートウェイ

プライベートサブネット内のEC2がインターネットにアクセスするための出口。

```
プライベートEC2 → NAT GW（パブリックサブネット）→ IGW → インターネット
```

- インターネットからの**受信は不可**（セキュリティ的に安全）
- パッチ適用・外部APIコールに使う
- 各AZに配置するのが推奨（冗長性確保）
- **有料**（時間課金＋データ転送量課金）

---

## セキュリティグループ

EC2などのインスタンスに付けるファイアウォール。

- **ステートフル**: インバウンドを許可したら、そのレスポンスのアウトバウンドは自動で許可
- 許可ルールのみ設定可能（デフォルトは全拒否）
- 複数のセキュリティグループをアタッチできる

```
例: Webサーバー用セキュリティグループ
  インバウンド: HTTP(80), HTTPS(443) → 0.0.0.0/0（全IP）から許可
  アウトバウンド: 全て許可（デフォルト）
```

```
例: DBサーバー用セキュリティグループ
  インバウンド: MySQL(3306) → Webサーバーのセキュリティグループからのみ許可
  アウトバウンド: 全て許可
```

---

## NACL (Network ACL)

サブネット単位で適用するファイアウォール。

- **ステートレス**: インバウンドとアウトバウンドを個別に設定が必要
- 許可・拒否の両方を設定できる
- ルールに番号があり、番号の小さい順に評価（最初にマッチしたルールが適用）

### セキュリティグループ vs NACL

| | セキュリティグループ | NACL |
|---|---|---|
| 適用範囲 | インスタンス単位 | サブネット単位 |
| ステート | **ステートフル** | **ステートレス** |
| ルール | 許可のみ | 許可・拒否の両方 |
| 優先度 | 全ルールを評価 | 番号順（最初にマッチしたもの） |
| 主な用途 | 日常的なアクセス制御 | 特定IPの緊急ブロックなど |

実務ではセキュリティグループをメインで使い、NACLは補助的に使う。

---

## VPCピアリング

異なるVPC同士をプライベートネットワークで繋ぐ。別アカウント・別リージョン間も可能。

```
VPC-A (10.0.0.0/16) ←ピアリング→ VPC-B (172.16.0.0/16)
```

- **推移的ルーティングは不可**: A↔B、B↔Cのピアリングがあっても A→C には通信できない
- 多数のVPCを繋ぐ場合は Transit Gateway が適している

---

## Transit Gateway

複数のVPCやオンプレ拠点を一元的に繋ぐハブ。

```
          Transit Gateway
         /     |     \
VPC-A  VPC-B  VPC-C  オンプレ(VPN)
```

VPCピアリングはN対N接続が複雑になるが、Transit Gatewayはスター型で管理しやすい。

---

## VPCエンドポイント

インターネットを経由せずにAWSサービス（S3、DynamoDBなど）に接続する。セキュリティ向上とコスト削減に。

| 種類 | 説明 |
|---|---|
| **ゲートウェイ型** | S3、DynamoDB向け（無料） |
| **インターフェイス型** | その他多くのAWSサービス向け（有料） |

```
プライベートEC2 → VPCエンドポイント → S3
（インターネット・NATゲートウェイを経由しない）
```

---

## VPN / Direct Connect

オンプレミスとAWSを繋ぐ方法。

| | Site-to-Site VPN | Direct Connect |
|---|---|---|
| 経路 | インターネット（暗号化） | 専用線 |
| 速度 | 最大1.25Gbps程度 | 最大100Gbps |
| 遅延 | 不安定になりやすい | 安定・低遅延 |
| コスト | 安い | 高い（物理回線） |
| 導入期間 | 数時間〜 | 数週間〜数ヶ月 |

## よく使う構成

### 標準的なWebアプリ構成

```
インターネット
    ↓ IGW
パブリックサブネット（AZ-a / AZ-c）
  ALB, NAT GW, 踏み台サーバー
    ↓
プライベートサブネット（AZ-a / AZ-c）
  EC2（アプリ）, RDS, ElastiCache
```

### 踏み台サーバー（Bastion Host）

プライベートサブネットのEC2にSSH接続するための中継サーバー。

```
開発者PC → SSH → 踏み台（パブリックサブネット）→ SSH → プライベートEC2
```

最近はSSM Session Managerを使うと踏み台なしでアクセスできる（セキュリティ的に推奨）。
