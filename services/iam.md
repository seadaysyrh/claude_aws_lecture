# IAM (Identity and Access Management)

## 概要

AWSリソースへのアクセスを制御するサービス。「誰が」「何を」「できるか」を定義する。AWSを使う上で最初に理解すべき最重要サービス。

---

## 主要コンポーネント

### ユーザー (User)

- AWSにログインする個人アカウント
- 人間が操作するためのもの
- アクセスキー（プログラムからのアクセス用）を発行できる

### グループ (Group)

- ユーザーをまとめる入れ物
- グループにポリシーを付けると、グループ内の全ユーザーに権限が適用される
- 個人ではなくグループ単位で権限管理するのがベストプラクティス

### ロール (Role)

- AWSサービスや外部ユーザーに権限を付与する仕組み
- パスワードやアクセスキーが不要（一時的な認証情報が自動で発行される）
- 例: EC2インスタンスにS3へのアクセス権を与えるロールをアタッチ

### ポリシー (Policy)

- 権限を定義したJSONドキュメント
- ユーザー・グループ・ロールにアタッチして使う

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject"],
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

---

## ポリシーの構造

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect":    "Allow" または "Deny",
      "Action":    "実行できるAPI操作（例: s3:GetObject）",
      "Resource":  "対象のAWSリソース（ARN形式）",
      "Condition": "追加条件（省略可）"
    }
  ]
}
```

### よく使うActionの例

| Action | 意味 |
|---|---|
| `s3:GetObject` | S3オブジェクトの読み取り |
| `s3:PutObject` | S3オブジェクトの書き込み |
| `ec2:DescribeInstances` | EC2インスタンス一覧の取得 |
| `lambda:InvokeFunction` | Lambda関数の実行 |
| `*` | すべての操作（ワイルドカード） |

### Conditionの例

```json
"Condition": {
  "IpAddress": {
    "aws:SourceIp": "203.0.113.0/24"
  }
}
```
→ 特定のIPアドレスからのアクセスだけ許可する

---

## ポリシーの種類

| 種類 | 説明 |
|---|---|
| **AWS管理ポリシー** | AWSが用意している既製品（AdministratorAccessなど） |
| **カスタマー管理ポリシー** | 自分で作るポリシー |
| **インラインポリシー** | ユーザー・ロールに直接埋め込むポリシー（管理が煩雑になりがちなので非推奨） |

---

## ポリシーの評価ロジック

複数のポリシーが競合する場合の判定順序。

```
1. 明示的な Deny → 即ブロック（最優先）
2. 明示的な Allow → 許可
3. デフォルト Deny → 許可なし＝拒否
```

**Denyは必ずAllowに勝つ。**

---

## 最小権限の原則

IAMの大原則。**必要な権限だけを付与する**。

- `AdministratorAccess` を何でもかんでも使わない
- 本番環境には絞った権限だけを付与
- 定期的に不要な権限を棚卸しする（IAM Access Analyzerが便利）

---

## ARN (Amazon Resource Name)

AWSリソースを一意に識別する文字列。ポリシーのResourceに使う。

```
arn:aws:s3:::my-bucket          ← バケット自体
arn:aws:s3:::my-bucket/*        ← バケット内の全オブジェクト
arn:aws:ec2:ap-northeast-1:123456789012:instance/i-xxxxxxxx
      ↑     ↑リージョン        ↑アカウントID      ↑リソースID
```

## よくある使い方

### EC2からS3にアクセスしたい

```
EC2インスタンス → IAMロール（S3読み取り権限） → S3
アクセスキーをEC2に置かない！ロールを使う！
```

### LambdaからDynamoDBを操作したい

```
Lambda実行ロール に DynamoDB操作ポリシーをアタッチ
→ Lambdaが自動で一時認証情報を取得して実行
```

### CIツールからデプロイしたい

```
GitHub Actions → OIDC連携ロール → ECR/ECS
（アクセスキーをGitHubに置くより安全）
```

## MFA (多要素認証)

AWSコンソールログイン時にワンタイムパスワードを追加要求する設定。ルートアカウントには必ず設定する。

MFAが必要な操作に制限するポリシーも書ける:

```json
"Condition": {
  "BoolIfExists": {
    "aws:MultiFactorAuthPresent": "true"
  }
}
```

---

## ルートアカウント

AWSアカウント作成時に使えるすべての権限を持つアカウント。

**日常利用はNG。** 以下の場合のみ使用:
- IAMユーザーの初期設定
- アカウント解約
- 課金設定の変更

ルートアカウントには必ずMFAを設定し、アクセスキーは発行しない。

---

## IAM Identity Center（旧SSO）

複数AWSアカウントへのシングルサインオン管理。複数アカウントを使う場合に便利。

```
開発者
  ↓ 1回ログイン
IAM Identity Center
  ├─ 開発アカウント（開発者権限）
  ├─ 本番アカウント（読み取り権限）
  └─ 管理アカウント（管理者権限）
```

---

## IAM Access Analyzer

設定した境界（AWSアカウント・組織）の外部からアクセスできるリソースを自動検出するツール。

- 意図せず外部公開しているS3バケット・IAMロールなどを発見できる
- セキュリティ監査に活用

---

## よくある失敗パターン

| 失敗 | 対策 |
|---|---|
| EC2にAdministratorAccessを付けてしまう | 必要なサービスだけ権限を絞る |
| アクセスキーをコードに直書きする | IAMロールを使う |
| ルートアカウントを日常使いする | IAMユーザーを作って使い分ける |
| アクセスキーを長期間ローテーションしない | 定期ローテーション or IAMロールに移行 |
