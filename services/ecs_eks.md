# ECS / EKS

## ECS (Elastic Container Service)

AWSが独自に開発したコンテナオーケストレーションサービス。Dockerコンテナをクラスターで管理・実行できる。

### 起動タイプ

| タイプ | 説明 |
|---|---|
| **Fargate** | サーバーレス。インフラ管理不要、コンテナ定義だけ書けばいい |
| **EC2** | EC2インスタンスを自分で管理するクラスターで動かす。細かい制御が必要なとき |

### 主要概念

```
クラスター
  └─ サービス（常時X台稼働を維持する）
       └─ タスク（コンテナの実行単位）
            └─ コンテナ（Dockerイメージから起動）
```

- **タスク定義**: コンテナの設定（イメージ、CPU/メモリ、環境変数など）をJSONで定義
- **サービス**: タスクを指定台数維持し続ける。ALBと連携してロードバランシングも可能

### Fargateのメリット

- EC2の管理（パッチ適用、スケーリング設定等）が不要
- タスクが使ったCPU/メモリ分だけ課金
- 小〜中規模ならFargateが手軽でおすすめ

---

## EKS (Elastic Kubernetes Service)

マネージドKubernetesサービス。Kubernetesのコントロールプレーン（マスターノード）をAWSが管理してくれる。

### ECSとの違い

| | ECS | EKS |
|---|---|---|
| オーケストレーション | AWS独自 | Kubernetes（OSS） |
| 学習コスト | 低い | 高い（K8sの知識が必要） |
| ポータビリティ | AWSに依存 | 他クラウドにも移行しやすい |
| 採用理由 | AWSに慣れている | 既存K8s資産がある、マルチクラウド |

### EKSの構成

```
EKS クラスター
  ├─ コントロールプレーン（AWSが管理）
  └─ ワーカーノード（EC2 or Fargate）
       └─ Pod（コンテナの最小単位）
```

### どちらを選ぶか

- **ECS**: AWSだけで完結、シンプルに使いたい → Fargateと組み合わせて楽に運用
- **EKS**: 既にKubernetes使っている、マルチクラウド対応が必要

---

## ECR (Elastic Container Registry)

ECS/EKSと一緒によく使うDockerイメージのレジストリ。Docker Hubの代わりにAWS内でイメージ管理できる。

```
開発PC → docker push → ECR → ECS/EKSがpullして起動
```
