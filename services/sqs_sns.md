# SQS / SNS

---

## SQS (Simple Queue Service)

フルマネージドなメッセージキューサービス。コンポーネント間のメッセージを非同期で受け渡す。

### 基本的な仕組み

```
プロデューサー → [キュー] → コンシューマー（Lambdaなど）
（メッセージを送る）  （メッセージを溜める）  （メッセージを取り出して処理）
```

### なぜキューを使うか

- **非同期処理**: 送信側が応答を待たなくていい（レスポンスが速くなる）
- **負荷平準化**: 突然の大量リクエストをキューで吸収してバックエンドへの負荷を均す
- **疎結合**: 送信側と受信側が直接依存しない

### キューの種類

| 種類 | 特徴 |
|---|---|
| **標準キュー** | 高スループット。順序は保証されない。少なくとも1回の配信（重複の可能性あり） |
| **FIFOキュー** | 順序を保証。重複排除機能あり。スループットは標準より低い |

### 主要設定

| 設定 | 説明 |
|---|---|
| **可視性タイムアウト** | コンシューマーがメッセージを処理中に他から見えなくする時間（デフォルト30秒） |
| **メッセージ保持期間** | 未処理メッセージの保持時間（デフォルト4日、最大14日） |
| **DLQ（Dead Letter Queue）** | 処理失敗したメッセージを別キューに退避 |

## SNS (Simple Notification Service)

Pub/Sub（発行/購読）モデルのメッセージング・通知サービス。1つのメッセージを複数の宛先に同時配信できる。

### 基本的な仕組み

```
パブリッシャー → [SNSトピック] → 複数のサブスクライバーに同時配信
                                  ├─ メール送信
                                  ├─ SMS送信
                                  ├─ SQSキューに送信
                                  ├─ Lambda起動
                                  └─ Webhook（HTTP/HTTPS）
```

### 主なユースケース

- アラート通知（CloudWatchアラーム → SNS → メール）
- 複数システムへのイベント配信（ファンアウトパターン）
- モバイルプッシュ通知

---

## SQSとSNSの使い分け

| | SQS | SNS |
|---|---|---|
| 配信先 | 1つのコンシューマー | 複数の宛先に同時配信 |
| 処理タイミング | コンシューマーがpullして処理 | pushで即時配信 |
| メッセージの保持 | コンシューマーが取り出すまで保持 | 保持しない（配信後は消える） |

### よく組み合わせるパターン（Fan-out）

```
SNSトピック
  ├─ SQSキューA → Lambda（処理A）
  ├─ SQSキューB → Lambda（処理B）
  └─ メール通知
```

SNSで一斉配信し、各SQSキューで独立して処理させる。

## EventBridge（旧CloudWatch Events）

より高機能なイベントバス。SQS/SNSより多くのAWSサービスと連携でき、イベントのルーティングも柔軟。

新規設計では EventBridgeを使うことが増えている。

---

## よく使うユースケース

- **SQS**: 注文処理の非同期化、画像リサイズジョブ、ワーカーキュー
- **SNS**: CloudWatchアラームの通知先、複数Lambdaへのイベント配信
- **SQS + SNS**: 大量イベントを複数のコンシューマーで並列処理
